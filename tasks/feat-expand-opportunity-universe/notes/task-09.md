# Task 09: Cross-Protocol Calculation Engine

- **Decisions**: `calculate_cross_protocol_spread` uses a single-cycle position (borrow `max_ltv * 0.9` of initial capital) rather than recursive looping. Cross-protocol risk is one position at two protocols — a second loop cycle would amplify risk past the point of usefulness for the MVP, and recursive loops are already handled by `simulate_looping` for same-protocol cases.
- **Decisions**: `_fetch_cross_protocol_opportunities` computes results on-the-fly from latest lending snapshots rather than relying solely on persisted `CrossProtocolCalculation` rows. The persisted rows are written by `trigger_cross_protocol_calculation` (orchestrator) but the route works without them, matching the pattern in `_fetch_loop_opportunities` which also re-computes when no calc row exists.
- **Decisions**: Scoring for cross-protocol is a simple `net_spread * 10.0`. No full ranker pipeline since cross-protocol opportunities don't have the same volatility/depth data as loop/carry. This is a deliberate simplification — add full ranker scoring when cross-protocol history accumulates.
- **Deviations**: `CrossProtocolCalculation` doesn't have a `safety_margin` column in the model (only `net_spread`, `leverage`, `risk_score`, `penalty_breakdown`). The route computes `safety_margin` inline and puts it in `strategy_details` only.
- **Trade-offs**: Could have normalized cross-protocol opps through the same `score_opportunities`/`rate_opportunities` pipeline as loop/carry. Skipped because cross-protocol lacks the same volatility/depth data and would require additional DB queries. Add when data matures.
- **Risks**: The `_fetch_cross_protocol_opportunities` function iterates `O(N^2)` over markets per asset — fine for the current scale (< 20 markets), but upgrade to a JOIN-based query if market count grows.
